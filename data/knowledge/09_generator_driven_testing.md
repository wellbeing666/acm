# 规格驱动的数据生成器策略

## 常见测试策略

- 大规模测试不应由模型直接输出完整 input，而应先生成 TestDataSpec，再生成本地 generator.py。
- TestDataSpec 描述测试类别、规模、参数范围、构造方法和合法性检查，不包含具体大规模输入。
- generator.py 应使用确定性随机种子，保证同一规格重复运行能得到一致数据。
- 每个测试类别对应一个或多个 `.in` 文件，文件名建议包含用例 id 和序号，例如 `TC003_01.in`。
- 生成器应覆盖小规模可人工检查数据、中等随机数据、大规模压力数据、针对错误算法的结构化反例。
- 对图论题，生成器应能构造链、星、环、稠密子图、多个连通块、重边或特殊边权分布。
- 对 DP 和贪心题，生成器应能构造局部最优陷阱、相等关键字、极端容量、转移边界。
- 对数论和字符串题，生成器应能构造质数/合数、周期串、重复串、边界长度等结构。

## 边界情况

- 生成器必须保证所有输出满足题面输入格式和约束。
- 对最大规模测试，避免在 Python 源码中硬编码巨大字符串，应算法式生成并流式拼接。
- 若题面有 `m <= min(n(n-1)/2, limit)`，生成边时必须避免超限和非法自环。
- 若题面不允许重复边、重复元素或空集合，生成器应显式去重或断言。
- 若题面允许多组数据，生成器应覆盖 `T=1` 和 `T` 较大两类。
- 输出目录必须只包含输入文件，不生成 `.out`、答案文件或标程。
- 大数据生成应控制运行时间和内存，避免构造过大的中间集合。

## 容易卡掉错误代码的反例类型

- 只使用随机生成，缺少链、星、完全不连通、几乎完全连通等结构化图。
- 生成器没有固定 seed，导致结果不可复现。
- 生成器把模型给出的抽象参数误当作原始输入，写出了不符合题面格式的数据。
- 生成器没有合法性断言，悄悄输出越界数据。
- 对最大规模图用全量 pair 集合采样，导致内存爆炸。
- 生成器输出了 expected_output 或 `.out` 文件，混淆了数据生成和答案生成职责。
- 大规模输入直接写在 JSON 或 prompt 中，导致 token 爆炸和模型截断。
